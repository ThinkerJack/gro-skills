# 开始做 build-it

## 核心原则

执行要跟着规格和计划走。没有计划时不要把普通编码伪装成执行；遇到事实变化，要暂停记录偏离原因并重新验证。
`build-it` 不自动提交代码；提交是显式交付动作。实现完成后先用 `prove-it` 验证，
用户明确要求提交时再用 `commit-it`。

## 什么时候用

- 用户已经确认要按已有规格或计划实现。
- 有清楚任务边界、完成标准和验证方式。
- 需要逐步修改代码、配置、文档或测试。
- 需要按仓库约定落地变更。

## 不要用于

- 还在讨论目标、需求或设计。
- 只有一句“改一下代码”，但没有边界、验收或验证方式；先用 `write-spec` 或 `make-plan`。
- 出现失败但原因不明；用 `fix-bug`。
- 实现已经完成，需要验收；用 `prove-it`。

## 输入

- 计划或明确任务。
- 规格和验收标准。
- 上游设计、规格、PRD、接口文档或仓库方案文档。
- 仓库规则、现有代码风格和测试命令。

## 上游文档处理

如果用户提供或仓库中已有上游设计 / 规格文档，先把文档当作执行输入真源处理：

- 先阅读相关文档，再读代码；不要只凭记忆或聊天摘要实现。
- 从文档抽出本轮执行清单、非目标、验收点和风险点。
- 文档足够具体时，直接按文档执行；文档只有设计没有步骤、且改动较复杂时，先用 `make-plan` 拆执行步骤。
- 实现中发现文档与代码现状、接口契约或用户最新要求冲突时，先记录冲突和建议处理；影响行为边界的冲突不要静默绕过。
- 因实现事实导致设计决策变化时，同步更新对应仓库文档，或在输出里明确要求回写。
- 不把上游文档当成无限授权；仍只做本轮用户要求的范围。

## 操作

1. 先定位当前计划步骤，确认只做这一段。
2. 阅读相关代码和项目规则，跟随现有模式。
3. 做最小必要改动，不提前实现后续步骤。
4. 对行为变化补充或更新相应验证；代码行为变化优先考虑测试先行。
5. 每完成一个逻辑步骤，运行对应检查并读输出。
6. 如果偏离计划，写明原因、影响和是否需要回到 `make-plan`。
7. 不顺手重构无关代码，不扩大 diff。

## 输出

- 代码或文档改动。
- 每步验证结果。
- 偏离计划说明。
- 剩余风险或未做事项。

## 停止条件

计划内改动完成，并且可以进入 `prove-it` 做完整验证；若用户要求提交，验证后交给
`commit-it`。

## 升级条件

- 发现计划错误：回到 `make-plan`。
- 发现规格缺口：回到 `write-spec`。
- 出现失败或异常：转 `fix-bug`。

## 成功证据

改动范围能逐项对照计划解释，没有计划外扩张，并且完成了与改动相匹配的局部验证。
