# 挑毛病 poke-holes

## 核心原则

重要决定不能只被支持，也必须被攻击。先暴露风险，比实现后返工便宜。

## 什么时候用

- 用户问“这个方案有没有问题”“帮我挑战一下”“别顺着我说”。
- 需求、设计、技术选型或实现计划即将进入下一阶段。
- 决策影响范围大、成本高、不可逆，或依赖强假设。
- 你感觉自己正在默认同意用户判断。

## 不要用于

- 用户只是要事实查证；用 `find-proof`。
- 用户目标本身还没说清楚；先用 `set-goal`。
- 已经有具体失败现象；用 `fix-bug`。

## 输入

- 被挑战的想法、需求、方案、计划或结论。
- 当前证据、约束和目标。
- 已知不可接受的风险。

## 操作

1. 先复述要挑战的主张，避免攻击错对象。
2. 找出隐含假设：用户、技术、时间、成本、数据、组织、维护性。
3. 列出主要失败模式。
4. 给出反例或替代解释。
5. 标注每个风险的严重度和不确定性。
6. 判断哪些风险需要证据，哪些可以通过设计或范围收缩处理。
7. 给出建议：继续、修改、暂停、找证据或拆小。

## 输出

- 被挑战主张。
- 隐含假设。
- 失败模式。
- 缺失证据。
- 替代方案或反例。
- 推荐动作和不确定性等级。

## 停止条件

高风险假设已经被接受、缓解、改写，或明确转入 `find-proof` / `make-design`。

## 升级条件

- 关键假设无证据：转 `find-proof`。
- 风险来自需求不清：转 `write-brief` 或 `set-goal`。
- 风险来自架构边界：转 `make-design`。

## 成功证据

输出里至少有一个“如果错了会怎样”的具体风险，而不是泛泛说“注意风险”。
