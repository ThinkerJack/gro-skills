# 定目标 set-goal

## 核心原则

先弄清楚要解决什么，再决定怎么做。目标不清楚时，任何计划、设计和代码都只是猜测。

## 什么时候用

- 用户说“我想做个东西”“帮我优化一下”“这个方向怎么样”。
- 任务听起来能做，但目标、范围、约束或成功标准不清楚。
- 用户想先讨论、梳理、想清楚，而不是立刻实现。
- 进入写需求、做设计或写代码前，还有关键未知项。

## 不要用于

- 用户已经给出清楚规格，只需要执行。
- 当前问题是测试失败、运行报错或线上异常；用 `fix-bug`。
- 目标清楚但需要证据支持；用 `find-proof`。

## 输入

- 用户原始请求。
- 已知背景、上下文、限制。
- 可能相关的项目规则或已有文档。

## 操作

1. 用一句话重述用户真正想解决的问题。
2. 列出已知条件、未知条件和你正在做的假设。
3. 把目标拆成结果目标和约束条件。
4. 定义可验收的成功标准。
5. 只在必要时问问题；优先问会改变方向的问题。
6. 给出最小下一步：继续澄清、找证据、写需求、做设计或直接执行。

## 输出

- 问题重述。
- 目标和非目标。
- 已知、未知、假设。
- 成功标准。
- 推荐下一步。

## 停止条件

目标已经清楚到可以进入 `write-brief`、`make-design`、`find-proof` 或执行阶段。

## 升级条件

- 发现用户目标自相矛盾：回到澄清。
- 发现关键判断没有依据：转 `find-proof`。
- 发现方案风险很高：转 `poke-holes`。

## 成功证据

你能用一句话说明“我们要解决什么”，并且成功标准可以被后续验证。
