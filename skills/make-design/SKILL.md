# 做设计 make-design

## 核心原则

设计先回答“怎么组织改动”，再回答“具体写哪几行代码”。好的设计让后续实现更小、更可验、更容易 review。

## 什么时候用

- 需求已经清楚，需要讨论技术方案或产品实现方案。
- 改动涉及多个文件、模块、状态、接口或数据流。
- 需要比较方案、取舍复杂度、风险和维护成本。
- 写可执行规格前，需要人类先审方案。

## 不要用于

- 需求还没定；先用 `write-brief`。
- 只需要小范围机械改动；可以直接 `make-plan` 或 `build-it`。
- 已经有具体失败；用 `fix-bug`。

## 输入

- 需求 brief。
- 现有代码结构和项目规则。
- 相关证据和约束。
- 可选方案或用户偏好。

## 操作

1. 简述推荐方案。
2. 定义模块、文件或系统边界。
3. 描述数据流、状态流、调用流或用户流。
4. 列出关键接口、数据结构和不变量。
5. 比较至少一个替代方案，并说明为什么不选。
6. 标注风险、未决问题和验证策略。
7. 明确哪些内容进入 `write-spec`，哪些留给实现判断。

## 文档沉淀规则

默认先在对话中输出设计，不要为了轻量问题制造仓库文档。满足任一条件时，应建议或直接创建仓库设计文档：

- 改动跨 2 个以上包、模块、服务或团队。
- 会影响后续多个任务、多人分工、接口契约或长期架构边界。
- 涉及关键数据流、状态流、启动流、鉴权、存储、路由、实时通信等系统性决策。
- 方案会作为 `write-spec`、`make-plan`、`build-it` 的输入。
- 用户明确要求整理、沉淀、归档、形成前置信息或给团队看。

创建文档前先说明落库理由和目标路径；用户没有指定路径时，优先使用：

- `docs/<版本>-<主题>技术方案.md`
- `docs/<主题>技术方案.md`

仓库设计文档必须包含：

- 背景与目标。
- 模块边界。
- 数据流、状态流或调用流。
- 关键决策与替代方案。
- 风险与验证策略。
- 未决问题。

## 下一步路由判断

输出设计后必须判断下一步，避免把方案直接误当实施计划：

- 设计很小、实现顺序明显、验收点清楚：建议直接进入 `build-it`。
- 涉及多个步骤、多个包、迁移、测试矩阵、多人协作或上线顺序：建议先进入 `make-plan`。
- 需要交给另一个 agent 或长时间后继续执行，且必须自包含：建议进入 `write-spec`。
- 需求目标、边界、接口事实或设计资源仍不清：回到 `write-brief` 或 `find-proof`。
- 方案风险高、取舍争议大或可能影响架构铁律：进入 `poke-holes`。

## 输出

- 方案概览。
- 模块和职责。
- 数据流或调用流。
- 关键决策和替代方案。
- 风险和验证策略。
- 未决问题。
- 若已创建仓库设计文档，给出文档路径和摘要；若未创建，说明未落文档的原因。
- 下一步建议：明确写出 `build-it`、`make-plan`、`write-spec`、`find-proof`、`write-brief` 或 `poke-holes` 之一，并给一句理由。

## 停止条件

方案足够让人类审阅，也足够让 `write-spec` 继续细化。

## 升级条件

- 发现需求缺口：回到 `write-brief`。
- 发现证据不足：转 `find-proof`。
- 方案风险较高：转 `poke-holes`。

## 成功证据

读者不看实现细节，也能理解改动边界、数据怎么走、风险在哪里。
