Gro Skills System v0.1

Gro Skills

让 AI 编程任务沿着清晰路径成长为可交付代码。

Gro Skills 是一套轻量 AI coding skill system。它的核心方法 GroPath 把模糊想法变成可验证交付，把关键决策留成证据，把项目差异留给使用方仓库。

12 core skills
2 public dirs
1 growth path

设计原则

核心只管通用路径和证据边界。技术栈、工具命令、公司流程和项目规则不塞进 core。

成长优先

每次任务都应让代码、上下文或流程变得更清楚，而不是只追求一次性完成。

证据优先

AI 不能只说“已完成”。关键结论、改动和验收都需要可检查证据。

轻核心

core skill 保持短小、通用、可组合，不接管所有工作细节。

少即是多

第一版只保留可执行 skills 和公开 docs；扩展目录等真的用到再加。

成长路径

这条路径不是每个小任务都要完整跑完。它定义了从模糊想法到可交付代码时，哪些动作应该存在。

定目标

先弄清楚问题、目标、约束和成功标准。

挑毛病

主动找反例、风险和隐藏假设。

找证据

先判断证据力度，再用代码、运行结果、文档、数据或网页支撑判断。

写需求

把想法变成边界清楚的需求说明。

做设计

决定结构、模块、数据流和技术取舍。

写规格

写成下一位 AI 可直接执行、验证和回流变更的详细说明。

拆计划

拆成小任务，每一步都有文件范围、验证方式和预期结果。

开始做

只在已有规格或计划时执行，偏离计划时说明原因。

修问题

失败时先复现、建假设，再最小修复。

验证它

跑测试、检查、截图或手动验收并修复失败。

做评审

审 diff、风险和证据，准备交付。

记经验

把重复踩坑、新规则和领域语言沉淀到长期上下文或自动化。

Core Skills v0.1

推荐使用短、好记、像命令的英文名。中文名用于文档和 README，英文名用于 skill 目录。

set-goal 定目标

intake

把模糊请求整理成清楚的问题、目标、约束和成功标准。

  • 输入：想法、任务、问题、用户意图。
  • 输出：目标重述、未知项、成功标准、下一步决定。
  • 停止：目标可被需求或设计使用。

poke-holes 挑毛病

challenge

攻击需求、方案或判断里的漏洞，防止 AI 顺从式赞同。

  • 输入：计划、需求、技术判断、产品假设。
  • 输出：失败模式、隐藏假设、反例、风险等级。
  • 升级：高风险且证据不足时回到找证据。

find-proof 找证据

evidence

判断证据力度，并从代码、运行结果、文档、数据或互联网里找到依据。

  • 输入：需要确认的问题或 claim。
  • 输出：证据、来源、可信度、缺口。
  • 分层：本地证据、运行证据、外部轻查、深度调研。
  • 边界：深度搜索、多引擎和归档放进 research pack。

write-brief 写需求

requirement

把目标和证据整理成可以被设计和实现使用的需求说明。

  • 输入：目标、用户场景、证据、约束。
  • 输出：需求、非目标、验收标准、开放问题。
  • 停止：需求足够进入设计。

make-design 做设计

design

在写代码前决定模块边界、数据流、关键取舍和风险。

  • 输入：需求、现有代码结构、约束。
  • 输出：方案概览、模块职责、数据流、风险。
  • 停止：人类可以审方案并做决定。

write-spec 写规格

spec

把已认可设计变成下一位 AI 不看原始资料也能执行、验证和回流变更的规格。

  • 输入：已认可设计和必要证据。
  • 输出：决定、文件级改动、接口、字段、测试契约。
  • 必须：包含 Verification 和 Change Policy。
  • 证据：每条关键要求能追溯到需求或设计。

make-plan 拆计划

plan

把规格拆成小而有序的实现任务，并为每步指定文件范围、验证方式和预期结果。

  • 输入：可执行规格。
  • 输出：任务列表、文件范围、依赖、每步验证和预期结果。
  • 停止：任务可被逐步执行和复查。

build-it 开始做

implement

执行已有规格或计划，遵守仓库约定，并记录计划外调整。

  • 输入：已确认计划、规格、仓库规则。
  • 输出：代码改动、局部验证、偏离说明。
  • 边界：不是普通编码入口；没有边界和验证先回到写规格或拆计划。

fix-bug 修问题

debug

失败出现时禁止猜测式修复，先复现、假设、验证，再最小改动。

  • 输入：失败现象、日志、测试结果、用户反馈。
  • 输出：复现路径、假设、证据、最小修复。
  • 红线：不复现不修复；修复后必须重跑原失败路径。

prove-it 验证它

verify

交付前运行验证和修复闭环，用本轮新鲜证据证明改动真的可用。

  • 输入：已完成改动、验收标准。
  • 输出：验证矩阵、失败分类、修复和复验结果。
  • 红线：没有新鲜验证证据，不能说完成、修好或通过。

review-it 做评审

review

交付前审查 diff、风险、测试证据和交接说明。

  • 输入：diff、规格、验证结果。
  • 输出：风险摘要、测试证据、提交或交接说明。
  • 边界：git/PR/发布细节可放入 agent-ops pack。

save-lesson 记经验

reflect

把重复失败、项目规则、领域语言和流程缺口沉淀到长期上下文。

  • 输入：任务结果、踩坑、反复出现的纠正。
  • 输出：发生了什么、是否可复用、应该写到哪里。
  • 去处：AGENTS、CONTEXT、ADR、测试、lint、脚本、pack 或 core。
  • 目的：让下一次任务站在这一次之上。

当前结构

第一版只保留正在使用的内容，避免空目录制造概念负担。

skills 通用路径

定义每个 AI coding 任务都可能需要的工作动作：定目标、找证据、写规格、验证、评审、沉淀。

docs 公开说明

只放一个介绍 Markdown 和一个 HTML 视图；历史调研与私有参考资料不进开源仓库。

运行规则

这些规则决定 Gro Skills 和普通提示词集合的区别。

小任务不强迫跑全链路

简单 bug 可以从 fix-bug 进入，简单修改可以从 make-plan 进入。build-it 只执行已有规格或计划。路径是地图，不是表演。

高风险边界必须有证据

需求、设计、规格、验证、评审和学习沉淀是关键边界。越接近交付，证据要求越高。

反对顺从式 AI

poke-holes 是一等公民。重要方案进入实现前，必须至少经历一次风险和假设检查。

经验必须回流

同一个纠正出现第二次，就应该进入 overlay、pack、core skill 或自动化检查，而不是继续靠记忆。

v0.1 已定口径

这些问题已经在第一版 core skill 里落成规则，后续使用中再按真实反馈迭代。

build-it 保留，但收窄

它只定义执行已有规格或计划时的纪律，不作为普通编码入口。这样保留执行阶段，又避免和 coding agent 默认能力重叠。

review-it 不包含 git/PR

v0.1 只做评审、风险判断和交接。commit、PR、merge、release 放进 agent-ops 或 git pack。

find-proof 只管证据分层

core 定义证据力度分层；research pack 负责深度搜索、中文优先、多源验证和调研归档。

frontmatter 用英文 id

skill 目录和 frontmatter `name` 用英文短名；文档和 README 保留中文别名，避免触发机制被中文别名稀释。

开源实践已经吸收进规则

Superpowers 给计划、执行、验证纪律；mattpocock/skills 给轻量组合和 CONTEXT/ADR 回流；OpenSpec/Spec Kit 给规格先于实现和可验证变更。
