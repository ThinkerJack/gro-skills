Gro Skills System v0.1

AI Coding 的双场景工作流

一组可执行的 AI coding skills：想清楚、写清楚、做出来、验证好。

Gro Skills 不接管你的开发流程。它只把 AI 编程里最容易翻车的几个关口固定下来：目标、证据、规格、计划、实现、验证、评审和复盘。

团队开发者 独立开发者 不强制全链路 证据优先
一句话用法

把你当前卡住的阶段说出来，让 AI 调用对应 skill：定目标、挑毛病、找证据、写规格、验证它。

先选你的场景

Gro Skills 的入口不是“背 12 个命令”，而是判断你现在处在团队交付还是独立项目推进里。

团队开发者：还原需求到交付代码

Team delivery

你已经有 PRD、设计稿、任务、接口或团队流程。Gro Skills 帮你把需求理解、方案、规格、实现和验收串起来，减少“AI 写得很快但写偏了”。

  • 澄清需求边界，避免漏掉非目标和验收口径。
  • 先做方案和可执行规格，再让 AI 写代码。
  • 交付前用本轮证据验证实现符合需求。
  • 评审时整理风险、diff、验证结果和交接说明。
write-brief -> make-design -> write-spec -> make-plan -> build-it -> prove-it -> review-it

独立开发者：从想法到可迭代项目

Solo project

你从想法、假设、灵感或一个市场观察开始，没有完整团队分工。Gro Skills 帮你先判断方向，再把每个版本变成可实现、可验证、可复盘的交付。

  • 把模糊想法磨成目标、约束和成功标准。
  • 主动挑毛病、找证据，避免做错方向。
  • 按版本写需求、做设计、拆计划和实现。
  • 把踩坑沉淀成自己的项目规则和自动化。
set-goal -> poke-holes -> find-proof -> write-brief -> make-design -> write-spec -> make-plan -> build-it -> prove-it -> save-lesson

新人怎么开口用

你可以直接用自然语言，不需要先记住 skill id。下面这些说法会把 AI 拉到正确阶段。

需求不够清楚 先别写代码，帮我把这个需求的目标、非目标、未知项和验收标准定清楚。
担心方案有坑 别顺着我说，先挑这个方案的问题，列出失败场景、隐藏假设和需要补的证据。
需要先查证 先找证据再判断。请区分代码证据、运行证据、文档证据和外部资料。
准备让 AI 实现 把这个方案写成可执行规格，要求下一位 AI 不打开原始资料也能实现和验证。
需要拆任务 按这个 spec 拆一个实现计划，每一步写清文件范围、验证方式和预期结果。
准备交付 验证这次改动。没有新鲜证据不要说完成，失败项先分类再修复和复验。

四个阶段

完整路径保留，但对外先讲阶段。小任务可以从中间进入，大任务才需要从头跑到尾。

想清楚

目标不清、证据不足、方案未经挑战时，不急着进入实现。

set-goal poke-holes find-proof

说清楚

把想法、证据和方案整理成需求、设计和可执行规格。

write-brief make-design write-spec

做出来

按计划实现；失败时系统化 debug，不猜着改。

make-plan build-it fix-bug

交付并成长

交付前验证和评审；重复教训回流成规则、文档或自动化。

prove-it review-it save-lesson

完整 Core Skills

英文 id 用于 skill 目录和触发机制；中文名用于理解和团队沟通。

set-goal 定目标

intake

把模糊请求整理成清楚的问题、目标、约束和成功标准。

  • 适合：想法早期、需求不清、范围不稳。
  • 输出：目标重述、未知项、成功标准、下一步。

poke-holes 挑毛病

challenge

攻击需求、方案或判断里的漏洞，防止 AI 顺从式赞同。

  • 适合：方案进入下一阶段前。
  • 输出：失败模式、隐藏假设、反例、风险等级。

find-proof 找证据

evidence

判断证据力度，并从代码、运行结果、文档、数据或网页里找到依据。

  • 适合：事实不确定、claim 需要验证。
  • 边界：深度搜索和归档进入 research pack。

write-brief 写需求

requirement

把目标和证据整理成可以被设计、实现和验收使用的需求说明。

  • 适合：团队任务入口、独立项目版本入口。
  • 输出：场景、需求、非目标、约束、验收标准。

make-design 做设计

design

在写代码前决定模块边界、数据流、接口、关键取舍和风险。

  • 适合：跨文件、跨模块、状态或接口复杂的改动。
  • 输出：方案、模块职责、数据流、决策和风险。

write-spec 写规格

spec

把已认可设计变成下一位 AI 不看原始资料也能执行和验证的规格。

  • 适合：准备交给 AI 实现、跨 session 交接。
  • 必须：Verification 和 Change Policy。

make-plan 拆计划

plan

把规格拆成有顺序、有依赖、有文件范围和验证方式的小任务。

  • 适合：实现前排任务、拆检查点。
  • 输出：任务列表、依赖、每步验证和预期结果。

build-it 开始做

implement

执行已有规格或计划，遵守仓库约定，并记录计划外调整。

  • 适合：已有清楚计划和验收标准。
  • 边界：不是普通“随便写代码”入口。

fix-bug 修问题

debug

失败出现时先复现、建假设、找证据，再做最小修复。

  • 适合：测试失败、运行报错、行为异常。
  • 红线：不复现不修复，修复后重跑失败路径。

prove-it 验证它

verify

交付前运行验证和修复闭环，用本轮新鲜证据证明改动真的可用。

  • 适合：准备说完成、修好、可交付前。
  • 红线：没有证据，不能声明通过。

review-it 做评审

review

交付前审查 diff、风险、测试证据和交接说明。

  • 适合：提交、PR、handoff 前。
  • 边界：git/PR/发布细节进入 pack 或 overlay。

save-lesson 记经验

reflect

把重复失败、项目规则、领域语言和流程缺口沉淀到长期上下文。

  • 适合：任务结束后、同类纠正第二次出现时。
  • 去处：AGENTS、CONTEXT、ADR、测试、lint、脚本或 pack。

什么时候不用全链路

路径是地图，不是表演。成熟用户应该从当前风险点进入。

小 bug

从 `fix-bug -> prove-it` 进入。先复现和定位，不需要补完整需求和设计。

明确小改

从 `make-plan -> build-it -> prove-it` 进入。边界和验收已清楚时，不要制造文档负担。

需求已经完整

从 `make-design` 或 `write-spec` 进入。团队已有 PRD 和设计稿时，不必重新做目标澄清。

想法还很早

从 `set-goal -> poke-holes -> find-proof` 进入。不要在问题还糊的时候让 AI 直接实现。

它和普通 prompt 有什么区别

Gro Skills 的价值不在“提示词更长”，而在把关键关口变成可复用流程。

普通 prompt

一次性说明你想让 AI 做什么。适合小问题，但很容易在长任务里丢边界、漏证据、跳过验证或被用户立场带偏。

Gro Skills

把工作拆成阶段，并规定每个阶段的输入、输出、停止条件和成功证据。AI 仍然灵活工作，但不能轻易跳过关键检查点。

运行原则

这些原则决定 Gro Skills 适合真实项目，而不是只适合演示。

证据优先

AI 不能只说“已完成”。关键结论、改动和验收都需要可检查证据。

反对顺从式 AI

重要方案进入实现前，至少经历一次风险、反例和隐藏假设检查。

保持简单

skills 只管通用动作和责任边界。具体技术栈、团队流程和项目命令留在使用方仓库。

经验必须回流

同一个纠正出现第二次，就应该进入长期上下文、测试、lint、脚本、pack 或 core skill。
